# RAG 知识库系统 — 分层实现计划

## 1. 项目初始化与基础架构搭建
- 初始化项目结构：`backend/`（FastAPI）和 `frontend/`（Vue3）目录分离
- 配置 `pyproject.toml` / `requirements.txt`，锁定依赖：FastAPI、uvicorn、SQLAlchemy[asyncio]、asyncpg、chromadb、ollama、python-multipart、python-jose、passlib、bcrypt
- 配置 `.env` 文件模板：数据库连接串、Ollama 地址、ChromaDB 存储路径、JWT 密钥
- 搭建 FastAPI 应用骨架：app factory、CORS 中间件、路由挂载、异常处理、日志配置
- 初始化 Alembic 迁移工具，连接 PostgreSQL

## 2. 数据模型与认证系统
- 设计 PostgreSQL 数据模型：
  - `User` 表（id, username, email, hashed_password, is_admin, created_at）
  - `KnowledgeBase` 表（id, name, description, category, owner_id nullable, is_global, created_at）— owner_id=null 表示管理员全局库
  - `Document` 表（id, kb_id, filename, file_path, file_type, chunk_count, status, created_at）
  - `ChatHistory` 表（id, user_id, kb_id, session_id, role, content, created_at）
- 实现 JWT 认证：登录/注册接口、token 签发与校验依赖（`get_current_user`）
- 实现角色权限中间件：admin 判定、全局知识库操作权限、用户只能操作自己的知识库
- Alembic 生成并执行首次迁移

## 3. 文档解析与向量化管道
- 实现统一文档解析器（`parser.py`）：
  - PDF → PyMuPDF / pdfplumber
  - DOCX → python-docx
  - XLSX → openpyxl
  - PPTX → python-pptx
  - TXT / MD / JSON / CSV / HTML → 直接读取 + BeautifulSoup（HTML）
- 实现文本分块策略：按段落/固定长度分块（chunk_size=512, overlap=64），保留元数据
- 实现 ChromaDB 集成：
  - 每个知识库对应一个 ChromaDB collection（以 kb_id 命名）
  - 通过 Ollama HTTP API (`http://localhost:11434/api/embeddings`) 调用 `qwen3-embedding:0.6b` 生成 embedding
  - 文档上传时：解析 → 分块 → 向量化 → 存入 ChromaDB + 更新 Document 记录
- 实现上传接口：`POST /api/documents/upload`（支持多文件，异步处理状态跟踪）

## 4. 知识库管理 API
- 知识库 CRUD：
  - `POST /api/kb` — 创建知识库（用户创建私有 / admin 创建全局）
  - `GET /api/kb` — 列出可访问的知识库（用户：自己的 + 全局的；admin：全部）
  - `GET /api/kb/{id}` — 知识库详情（含文档列表）
  - `DELETE /api/kb/{id}` — 删除知识库（级联删除文档 + ChromaDB collection）
  - `PUT /api/kb/{id}` — 更新知识库信息
- 文档管理：
  - `GET /api/kb/{id}/documents` — 文档列表
  - `DELETE /api/documents/{id}` — 删除文档（同步删除向量）
  - `POST /api/documents/{id}/reindex` — 重新向量化
- 分类管理：
  - `GET /api/categories` — 获取所有分类
  - `POST /api/categories` — admin 添加分类

## 5. RAG 对话核心
- 实现检索管道：
  - 用户输入 → Ollama embedding → ChromaDB similarity search（top_k=5）
  - 结果过滤：只检索用户有权访问的知识库对应的 collection
- 实现对话接口：
  - `POST /api/chat` — 接收 query + kb_ids[]，拼接 prompt template（系统提示 + 检索到的上下文 + 用户问题）
  - 调用 Ollama API (`http://localhost:11434/api/chat`) 使用 `qwen3.5:2b` 生成回答
  - 流式返回（SSE / StreamingResponse）
  - 记录对话历史到 PostgreSQL（`ChatHistory` 表）
- `GET /api/chat/history` — 获取历史对话列表
- `GET /api/chat/history/{session_id}` — 获取某次对话详情

## 6. 前端界面开发
- 技术选型：Vue3 + Vite + Element Plus
- 页面规划：
  - **登录/注册页** — 表单验证、JWT 存储
  - **知识库管理页** — 列表展示（区分全局/个人）、创建/编辑/删除、分类筛选、上传文档（拖拽上传 + 进度条）
  - **对话页** — 知识库选择器（多选）、对话界面（流式打字效果）、历史对话列表、Markdown 渲染
  - **管理员面板**（admin only）— 用户管理、全局知识库管理、系统统计
- API 对接层：封装 axios，统一错误处理，token 自动刷新
- 响应式布局，支持移动端基本访问

二、需要修改的点（4处）
修改1：数据模型 — 拆分 ChatHistory
你的计划里 ChatHistory 是一张表，既存会话又存消息。建议拆成两张表：
# 原设计：
ChatHistory（id, user_id, kb_id, session_id, role, content, created_at）
# 问题：session_id 不是主键，查"某个用户的所有会话列表"很别扭
# 修改后：
# 1. 会话表（一次完整的对话）
class Conversation(Base):
    id          = Column(UUID, primary_key=True)  # 用UUID，防止被遍历
    user_id     = Column(Integer, ForeignKey("users.id"))
    title       = Column(String)                  # 自动取第一条消息前20字
    kb_ids      = Column(ARRAY(Integer))          # 关联的知识库ID列表
    created_at  = Column(DateTime)
# 2. 消息表（每条聊天记录）
class Message(Base):
    id              = Column(UUID, primary_key=True)
    conversation_id = Column(UUID, ForeignKey("conversations.id", ondelete="CASCADE"))
    role            = Column(String)     # user / assistant / system
    content         = Column(Text)
    sources         = Column(JSONB)      # ← 新增！存引用来源
    created_at      = Column(DateTime)
为什么改：
查会话列表：SELECT * FROM conversations WHERE user_id=1，一条 SQL 搞定
删会话：DELETE FROM conversations WHERE id=xxx，CASCADE 自动删消息
sources 字段：存 AI 回答引用了哪些文档段落，前端可以展示
修改2：知识库与 ChromaDB 的映射关系
原计划："每个知识库对应一个 ChromaDB collection（以 kb_id 命名）"
建议改为：用一个大 collection + metadata 过滤
# 原方案：每个知识库一个 collection
collection_name = f"kb_{kb_id}"  # 100个知识库 = 100个collection
# 修改方案：统一一个 collection，用 metadata 区分
collection_name = "all_documents"
# 每条向量带 metadata: {"kb_id": 1, "doc_id": 5, "chunk_index": 3}
为什么改：
跨知识库检索时，一个 collection 查一次就行，不用查多个再合并
删知识库时：collection.delete(where={"kb_id": 1})，一样干净
Collection 太多时 ChromaDB 管理开销大
修改3：对话接口增加 sources 返回
原计划只返回 AI 回答，建议同时返回引用的原文段落：
# 后端返回结构
{
    "answer": "根据文档规定，年假为5天...",
    "sources": [
        {
            "doc_id": 1,
            "filename": "员工手册.pdf",
            "page": 3,
            "content": "正式员工入职满1年后，享有5天带薪年假...",
            "score": 0.87
        }
    ]
}
前端展示效果：
🤖 AI：根据文档规定，年假为5天...
      📎 引用来源：员工手册.pdf 第3页 [展开原文]
修改4：文档处理改为异步任务
原计划"上传时解析→分块→向量化"，如果文件大会阻塞请求。建议后台任务：
# 原方案：同步处理，上传接口等处理完才返回
@router.post("/upload")
async def upload(file):
    text = parse(file)        # 慢！PDF可能要10秒
    chunks = split(text)      # 慢！
    embed_and_store(chunks)   # 慢！Embedding要调模型
    return {"status": "ready"}  # 用户等了30秒才看到响应
# 修改方案：上传后立即返回，后台异步处理
@router.post("/upload")
async def upload(file):
    # 1. 存文件，状态标为 processing
    doc = Document(status="processing")
    db.add(doc)
    await db.commit()
    # 2. 丢给后台任务
    background_tasks.add_task(process_document, doc.id, file_path)
    # 3. 立即返回
    return {"status": "processing", "doc_id": doc.id}
async def process_document(doc_id, file_path):
    try:
        text = parse(file_path)
        chunks = split(text)
        await embed_and_store(chunks)
        # 更新状态为 ready
        doc.status = "ready"
        doc.chunk_count = len(chunks)
    except Exception as e:
        doc.status = "failed"
        doc.error = str(e)
前端可以轮询状态或WebSocket 推送：
上传中... ██████████ 100%
处理中... 解析文档 ✓  切分文本 ✓  向量化 ■■■□□ 60%
处理完成 ✓
三、需要新增的点（3处）
新增1：健康检查与依赖确认接口
@router.get("/api/health")
async def health_check():
    """前端启动时调用，确认所有服务可用"""
    checks = {
        "database": await check_db(),        # PostgreSQL 连接
        "chromadb": await check_chroma(),    # ChromaDB 连接
        "ollama_llm": await check_ollama("qwen3.5:2b"),
        "ollama_embed": await check_ollama("qwen3-embedding:0.6b"),
    }
    all_ok = all(checks.values())
    return {"status": "ok" if all_ok else "degraded", "services": checks}
用处：前端页面加载时如果 Ollama 没启动，直接提示用户，而不是等到提问时才报错。
